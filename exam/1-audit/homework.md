### Подробнее про auditctl 
> В общем виде любое правило выглядит как:
> ```
> sudo auditctl -a ACTION,LIST -S SYSCALL -F FIELD=VALUE -k KEY
> ```
> где:
> - -a - действие
>   - always - всегда логировать
>   - never - исключить (зачем?)

`never` используется для поддавления генераций событий, чтобы не нагружать память диска, например:
```
sudo auditctl -a never,task
```
останавливает логирования частых системных вызовов, таких как `fork(), clone()`.

> - LIST:
>   - exit - фиксировать выход
>   - entry - фиксировать вход (зачем?)

В моей версии ядра 6.8.0-100-generic параметр `entry` не поддерживается.
Зачем это было надо? Допустим есть правило:
```
sudo auditctl -a exit,never -S read -F comm=nginx
```
т.е. не логируем вызовы от read от процесса nginx.
- При использовании entry: Система видит вызов read от nginx прямо на входе и моментально забывает о нем.
- При использовании exit: Ядру приходится создавать "контекст аудита", следить за выполнением вызова, тратить память и такты процессора, и только в самом конце, перед выходом, проверить правило и понять: "А, это же nginx, удаляем данные".
Почему это не нужно сейчас? Механизмы управления контекстами в ядре стали настолько быстрыми, что эта разница в микросекундах стала незначительной по сравнению с удобством использования только одного списка.

> Зачем нужен флаг -w и что за предупреждение пишет auditctl, если использовать этот флаг?

Добавляет правило мониторинга файлового объекта. Оно указывает ядру мониторить все доступы (чтение, запись, выполнение, изменение атрибутов) к указанному файлу или директории..
Пример:
```
sudo auditctl -w /etc/shadow -p rwxa -k shadow_access
```
- -w /etc/shadow: отслеживать файл /etc/shadow.
- -p rwxa: отслеживать операции Read, Write, Execute, Attribute change.
- -k shadow_access: добавить метку shadow_access для удобного поиска в логах.

Часто выводит предупреждение: "Use of syscall rules is more efficient than watch rules".
Во время выполнения команды ядро ставит метку на файл, если файлов много, то ядру придётся тратить больше ресурсов на отслеживание каждого индивидуального узла.

> 2.1 Создайте правило, которое логирует выполнение ls

```
sudo auditctl -w /usr/bin/ls -p x -k ls
```

> 2.2 Создайте правило, которое логирует изменение файла /etc/shadow ИЛИ изменение атрибутов этого же файла

```
sudo auditctl -w /etc/shadow -p wa -k shadow_access
```

> 2.3 Создайте правило, которое логирует неудачное завершение программы

```
sudo auditctl -a always,exit -F arch=b64 -S exit_group -F "exit!=0" -k app_error
```

> 2.4* Создайте правило, которое логирует удаление файлов (подумайте, какой системный вызов отвечает за удаление)

Используются системные вызовы unlink/unlinkat:
```
sudo auditctl -a always,exit -F arch=b64 -S unlink -S unlinkat -k file_rm
```
> 2.5* Создайте правило, которое логирует создание новых файлов (что происходит при создании файла?)

Вызываются системные вызовы open/openat:
```
sudo auditctl -a always,exit -F arch=b64 -S open -S openat -k file_create
```
> 2.6 Создайте правило, которое логирует выполнение программ только пользователем с UID 1000

Решения типа:
```
sudo auditctl -a always,exit -F arch=b64 -S all -F "uid=1000" -k uid_1000
```
это классический пример того, как можно случайно устроить системе DoS-атаку (отказ в обслуживании) или просто сделать логи абсолютно бесполезными.
Вот основные причины, почему использовать -S all вместе с фильтром по пользователю — это плохая идея:

1. Катастрофическое падение производительности
Системный вызов (syscall) происходит практически при каждом действии программы. Когда вы пишете -S all, ядро должно проверять каждое малейшее действие пользователя UID 1000:
- Чтение каждого байта из файла (read).
- Запись каждого байта в терминал или файл (write).
- Проверка времени (gettimeofday).
- Любое движение мышкой в графическом интерфейсе или выделение памяти (mmap, brk).

Даже при простом движении курсора или открытии браузера генерируются тысячи системных вызовов в секунду. Ядро будет тратить огромную часть ресурсов процессора только на то, чтобы записать эти действия в лог. Система начнет заметно "тормозить".

2. Лавинообразный рост логов (Disk Space)
Лог-файл /var/log/audit/audit.log переполнится за считанные минуты. 
- Одна простая команда ls порождает около 100+ системных вызовов.
- Запуск браузера породит сотни тысяч записей.
- Ваш жесткий диск быстро забьется гигабайтами мусорной информации, в которой невозможно найти что-то действительно важное.

Вместо того чтобы следить за всеми вызовами, нужно следить только за значимыми:
```
sudo auditctl -a always,exit -F arch=b64 -S execve -S execveat -F uid=1000 -k user_1000
```

> 2.7 Создайте правило, которое логирует только неуспешные попытки открытия файла /etc/passwd
```
sudo auditctl -a always,exit -F arch=b64 -S open -S openat -F "path=/etc/passwd" -F "exit!=0" -k passwd_failure
```

> 2.8* Создайте правило, которое логирует все случаи, когда пользователь с UID 1000 пытается изменить файл, принадлежащий другому пользователю
```
sudo auditctl -a always,exit -F perm=wa -C "uid!=obj_uid" -k not_owner_access
```

> 2.9* Создайте правило, которое фиксирует изменение прав доступа к файлам (chmod), но только если новые права делают файл исполняемым
```
sudo auditctl -a always,exit -F arch=b64 -S chmod -F perm=a -F a1&0111 -k making_executable
```

> 2.10* Создайте правило, которое фиксирует все запуски программы любым пользователем кроме root
```
sudo auditctl -a always,exit -F arch=b64 -S execve -F euid!=0 -k not_root
```
